#engineering

Las historias de usuario son una técnica ágil para capturar requisitos de forma simple y centrada en el usuario.

## Formato

```
Como [rol], quiero [funcionalidad], para [beneficio].
```

## Ejemplo

```
Como usuario registrado, quiero recuperar mi contraseña por correo,
para poder acceder a mi cuenta si la olvido.
```

## Criterios de Aceptación

Cada historia de usuario debe tener criterios de aceptación claros:

```
Criterios de aceptación:
- El usuario debe recibir un correo con enlace de recuperación
- El enlace debe expirar después de 24 horas
- El usuario debe confirmar la nueva contraseña dos veces
```

## Ventajas

- Simples y fáciles de entender
- Centradas en valor para el usuario
- Fáciles de estimar y priorizar
- Permiten iteraciones rápidas

## Limitaciones

- No capturan requisitos no funcionales
- Pueden ser ambiguas sin criterios claros
- No suited para requisitos complejos
